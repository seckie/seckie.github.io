このブログの記事は Markdown ファイルを GitHub リポジトリで管理していて、main ブランチに push すると GitHub Actions で自動デプロイされます。仕組みはシンプルなのですが、記事を書き始めるたびにブランチを切って、frontmatter を書いて、PR を作って……という一連の手作業が地味に面倒でした。そこで [Claude Code](https://code.claude.com/) のスキル機能を使って、このワークフローを自動化してみました。

## スキル機能でワークフローを定義する

Claude Code には「スキル」という機能があり、リポジトリの `.claude/skills/` 以下に Markdown で手順書を置いておくと、Claude Code がそれを読んで一連の作業を実行してくれます。

今回は「新しい記事を書きたい」と伝えるだけで、次の流れが自動で進むスキルを作りました。

1. トピックからスラッグ(URL になる英語のファイル名)を決める
2. `src/content/blog/` に frontmatter 付きの Markdown ファイルを作成し、本文の下書きを生成する
3. `npm run build` で content collection のスキーマ検証を通す
4. 専用ブランチにコミットして push する
5. Draft PR を作成する

手順書には frontmatter の書式(日付フォーマットやタグの形式)や、既存記事の文体に合わせることまで書いてあるので、生成される下書きがこのブログの体裁から外れることはありません。あとは PR のブランチ上で本文を仕上げるだけです。

## ブランチは post/スラッグ 形式に

記事ごとのブランチ名は `post/<スラッグ>` という形式に統一しました。たとえばこの記事なら `post/claude-code-blog-workflow` です。

- ブランチ名を見るだけで「どの記事の作業か」が分かる
- `post/` プレフィックスで記事ブランチとその他の作業ブランチを区別できる
- スラッグはそのまま記事の URL(`/blog/<スラッグ>/`)になるので、ブランチ・ファイル名・URL が一対一で対応する

命名規則をスキルの手順書に書いておけば、毎回同じルールでブランチが切られるので、自分で覚えておく必要がないのも良いところです。

## Draft PR で書きかけ記事の誤公開を防ぐ

このブログは main ブランチにマージした瞬間に本番へデプロイされます。つまり、書きかけの記事の PR をうっかりマージしてしまうと、そのまま公開されてしまいます。

そこでスキルでは、PR を必ず **Draft PR** として作成するようにしました。Draft のままではマージボタンが押せないので、「書きかけのままマージして誤公開」という事故が仕組みとして起きません。記事を書き上げてプレビューを確認したら、Ready for review に切り替えてマージする、という運用です。

PR の本文にも「仕上げる → プレビュー確認 → Ready for review にしてマージ」というチェックリストを自動で入れているので、公開までの手順に迷うこともありません。

## まとめ

- Claude Code のスキル機能で、記事テンプレート作成から Draft PR 作成までを自動化した
- ブランチは `post/<スラッグ>` 形式に統一して、記事と作業ブランチを一対一で対応させた
- PR を Draft にすることで、書きかけ記事の誤公開を仕組みで防いでいる

「書き始めるまでの準備」がゼロになると、記事を書くハードルが目に見えて下がります。ちなみにこの記事自体も、このワークフローで作られた下書きから書き始めたものです。
