このブログの記事は Markdown ファイルとして GitHub リポジトリで管理していて、main ブランチに push すると GitHub Actions が自動でビルド・デプロイする構成になっています。シンプルで気に入っているのですが、新しい記事を書き始めるたびに「ブランチを切って、frontmatter 付きのテンプレートファイルを作って、コミットして、PR を作る」という同じ手順を毎回繰り返していました。

この定型作業を [Claude Code](https://claude.com/claude-code) のスキル機能で自動化してみたら快適だったので、その話を書きます。

## スキル機能とは

Claude Code には「スキル」という仕組みがあり、リポジトリの `.claude/skills/` ディレクトリに手順を書いた Markdown を置いておくと、スラッシュコマンドのように呼び出せます。プロジェクト固有のワークフローを自然言語で定義しておけるのがポイントで、シェルスクリプトを書くほどかっちり固めなくても、「こういう手順でやってほしい」を伝わる形で残しておけます。

## 自動化した内容

記事作成用のスキルを呼び出すと、次の一連の流れを Claude Code がやってくれます。

1. 記事タイトルから URL スラッグを決める
2. `post/スラッグ` 形式で新しいブランチを作成する
3. `src/content/blog/スラッグ.md` に frontmatter 入りの記事テンプレートを生成する
4. 変更をコミットして push する
5. GitHub に **Draft PR** を作成する

あとは生成されたテンプレートに本文を書いていくだけです。書き上がったら Draft を外してマージすれば、GitHub Actions が公開まで面倒を見てくれます。

## ブランチ名は post/スラッグ 形式に

ブランチ名は `post/スラッグ` という形式に統一しました。たとえばこの記事なら `post/automating-blog-posts-with-claude-code` です。

命名規則を決めておくと、ブランチ一覧を見ただけで「これは記事のブランチだ」と一目でわかりますし、機能開発用の `feature/` ブランチと混ざりません。規則が機械的に決まるということは、スキルに任せやすいということでもあります。

## Draft PR で書きかけ記事の誤公開を防ぐ

このワークフローで一番気に入っているのが、PR を最初から **Draft** として作る点です。

このブログは main ブランチへの push がそのまま本番デプロイにつながるので、書きかけの記事がうっかりマージされると、未完成の記事がそのまま公開されてしまいます。Draft PR にしておけば GitHub 上でマージボタンが無効化されるため、「勢いでマージしてしまった」という事故が仕組みとして起こりません。

「執筆中 = Draft、公開したい = Ready for review に変更してマージ」という運用は、記事のステータス管理としてもわかりやすくて気に入っています。

## まとめ

- Claude Code のスキル機能で、記事テンプレート作成から Draft PR 作成までを自動化した
- ブランチは `post/スラッグ` 形式に統一して、記事ブランチを見分けやすくした
- Draft PR にすることで、書きかけ記事の誤公開を仕組みで防いでいる

定型作業が「スキルを呼び出して記事名を伝えるだけ」になったので、記事を書き始めるまでの腰の重さがだいぶ減りました。ブログのように「始めるまでの手数」がモチベーションに直結する趣味では、この手の自動化は効果が大きいと感じています。
